### Ход работы 
В певой лабораторной работе построена сервисная модель AWS.Классификация выполнялась по признакам из биллинга (Product Code, Usage Type, Operation и описания строки). Например, по Product Code определялся сервис (AmazonS3, AmazonEC2, AmazonRedshift), после чего строки раскладывались по иерархии IT Tower, Service Family,Service Type,Service Sub Type,Service Usage Type.

Важно отметить, что одинаковые правила разбиения применяются ко всем данным и сохраняются при переходе к новому провайдеру. Поэтому во 2-й лабораторной верхний уровень IT Tower и семейства Service Family выбирались из образца и согласовывались с логикой лабы 1.

В лабораторной работе мыслили также, определяли сервис Azure по полям Consumed Service и Meter Category. Сверялись с документацией и смыслом метра: Meter Name и Meter Sub-Category подсказывали, за что именно берут деньги.

Заполняя таблицу пытались ответить на вопросы: что за сервис по смыслу и За что именно списывают деньги в этой строке, какая метрика. 
Всё также у нас есть: 
* IaaS - инфраструктура как сервис (пример: Azure Virtual Machines, Azure Firewall)
* PaaS - платформа как сервис (пример: Azure Data Factory, Azure Database for PostgreSQL/MySQL)
* SaaS - ПО как сервис

#### Если смотреть то, что уже было дано:
Consumed Service часто прямо указывал на продукт (допустим Microsoft.DataFactory)
Meter Category обычно подсказывает крупную тему (CDN, Data Factory, Redis Cache)
Meter Name и Meter Sub-Category дают детали тарификации (Data Transfer) 

**Первый столбец IT Tower** заполнялся по логике: мысленно поднимались на самый высокий уровень и решали: это инфраструктура, хранилище, база данных, сеть или управляемые облачные сервисы. (допустим видим что в Costumer Service написано Microsoft.AnalysisServices - это управляемомый аналитический сервис Microsoft, Analysis Services- это не виртуальная машина и не железо, которое мы настраиваем сами. Это готовый облачный продукт для аналитики. Сверяем с лабой 1 Analytics у нас была Service Family внутри Cloud Services, а не отдельный IT Tower.
Значит для Azure Analysis Services мы делаем так же.)

**Service Family** - семейство выбирали по смылу: Analytics – если сервис про анализ данных, интеграцию, BI (Data Factory, Analysis Services). Security and Identity – если сервис про ключи, SIEM (Key Vault, Sentinel). Management Tools – если сервис про управление/оркестрацию (Scheduler).

В столбце **Service Type** - на этом уровне мы давали сервису понятное читаемое имя. (смотрели на Consumed Service)

В столбце **Service Sub Type** - он должен отражать вариант сервиса. (допустим в Azure Analysis Services: В биллинге встречались уровни Basic/Developer/Standard, поэтому Sub Type = Basic или Developer или Standard. Это разные уровни сервиса, их важно различать при анализе стоимости.)

**Service Usage Type** - за что берут деньги. (например Azure Key Vault: Usage Type = Transactions, потому что тарификация строится на количестве операций с секретами/ключами.)
